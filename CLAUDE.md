# PetPlanWise — Project Guide for Claude

Static HTML pet cost calculator site for U.S. dog and cat owners. Live at
**https://petplanwise.com**. No build step, no framework, vanilla JS, deployed
on Vercel from `main`. Repo: `github.com/mlanon7/petplanwise`.

---

## Architecture in one paragraph

Pure static HTML/CSS/vanilla JS. **No build, no bundler.** Every page is hand-
authored HTML (or generated by a Node script in `scripts/`). Numeric data lives
in `assets/data/csv/*.csv` and is loaded by `csv-loader-<VERSION>.js`. The
calculator engine in `calculator-<VERSION>.js` reads those CSVs and renders
prefilled estimates. The site auto-deploys from the `main` branch on push;
Vercel serves files directly with no transformation.

## The cache-bust pattern (read this twice)

Every CSS/JS asset is renamed on each deploy to a date-keyed filename:

```
assets/css/site.css                  # served as site.css?v=<VERSION>
assets/js/layout-<VERSION>.js
assets/js/calculator-<VERSION>.js
assets/data/csv-loader-<VERSION>.js
```

Current version lives in `scripts/bump-cache-bust.js` as `NEW_V`. Format
`YYYYMMDD<letter>` (e.g. `20260516w`). Letter increments each bump within a day.

**To bump cache (after any CSS/JS change):**

1. Edit `scripts/bump-cache-bust.js`: update `OLD_V`, `NEW_V`, the `RENAMES`
   array, and the `replacements` array (all 4 places must match).
2. Run `node scripts/bump-cache-bust.js`.
3. The script renames the three JS files and rewrites every `?v=<OLD>` → `?v=<NEW>`
   in every HTML file. It also force-updates any `hero.{jpg,svg,png,webp}?v=*` to
   the new version (so swapped hero photos invalidate CDN cache automatically).

**Why this matters:** Vercel caches `/assets/*` with `Cache-Control: immutable,
max-age=31536000`. Without a new URL, browsers and the CDN keep the old file
for a year. This bit us multiple times during development.

## The CSV-first data model

Edit numbers in `assets/data/csv/`, not in JS files:

| File | Drives |
|---|---|
| `breeds.csv` | Per-breed name, species, size, grooming/health multipliers, purchase ranges, lifespan, notes, source_url, last_reviewed |
| `breed-traits.csv` | Weight, height, energy, alone-hours, kid/stranger friendliness, trainability, shedding, top facts (pipe-delimited), C-BARQ-style temperament scores |
| `breed-images.csv` | Per-breed hero src + alt + credit + license + width/height |
| `base-costs.csv` | Annual base ranges per category (food, vet, grooming, etc.) |
| `procedures.csv` | Procedure cost ranges (spay/neuter, dental, surgery, etc.) |
| `state-multipliers.csv` | State cost adjustments |
| `city-multipliers.csv` | City-level adjustments (e.g. Manhattan +18%) |
| `insurance-monthly-premium.csv` | NAPHIA-aligned premiums by species/stage |
| `life-expectancy.csv` | Default lifespan by species + size |
| `reviewer.csv` | Editorial reviewer attribution |

All cost CSVs now have `source_url` + `last_reviewed` columns. Don't drop them.

## Calculator engine model

`calculator-<V>.js` computes annual cost as a product of multipliers:

```
annual = base × sizeMult × ageMult × lifestyleMult × locationMult × breedMult
```

**Lifetime is PHASE-WEIGHTED, not stage-multiplied.** Selecting `senior` does
NOT project senior-level annual costs across the full lifespan. Instead the
engine computes annual cost separately for puppy/kitten + adult + senior years
and sums them. Don't break this — it's the biggest accuracy win we shipped.
The regression test for it lives in `tests/calculator.test.js`.

## Page authoring conventions

**Every breed page has this structure:**

1. `<head>` with meta + JSON-LD (BreadcrumbList + FAQPage)
2. Hero figure with `<picture>` wrapping WebP + JPG fallback
3. `breed-snapshot` block — 1-sentence intro + 4 trait chips (weight, energy, kid, alone-hours)
4. Calculator widget (`<div data-calculator="dog" data-breed="slug" data-stage="adult">`)
5. Cost summary table with visible source-notes block below
6. Breed-specific cost drivers (4 items)
7. Insurance section
8. Ways to save (4 items)
9. Affiliate disclosure + Editorial recommendation CTA
10. FAQ (3 items, mirrored in JSON-LD)
11. Reviewer block
12. Sources list
13. Compare CTA
14. Full traits card

**Generators in `scripts/`** can produce new breeds wholesale — see
`add-twelve-breeds.js` and `add-five-more-breeds.js` for templates. Don't
hand-write a breed page if you're adding more than one.

## Hero photo handling (THIS HAS A SHARP EDGE)

Each breed directory has:

```
hero.jpg          # ~200 KB optimized progressive JPEG (1600px wide)
hero.webp         # ~140 KB WebP at same dims (modern browsers fetch this)
hero-original.jpg # FULL-quality source; used by optimize-breed-heroes.js to re-encode
hero-prev.jpg     # Sometimes present from prior swaps (safe to leave)
credit.json       # Pexels/Unsplash source URL, artist, license, description
```

**The gotcha:** `optimize-breed-heroes.js` uses `hero-original.jpg` as its
source of truth. If you swap `hero.jpg` without also overwriting
`hero-original.jpg`, the next optimize run silently reverts your swap.
**Always write BOTH files when changing a hero photo.** See
`scripts/swap-corgi-boston-heroes.js` for the correct pattern.

**Compare page + Find My Breed page reference `hero.jpg` via JS.** These two
pages had bare URLs without cache-busters, so swapped photos didn't refresh on
the CDN. Both now use `hero.jpg?v=<VERSION>`. The cache-bust regex sweep in
`bump-cache-bust.js` auto-updates them.

## Deploy + versioning workflow

The expected loop for ANY content/code change:

1. Make the change
2. (If you touched JS/CSS) `node scripts/bump-cache-bust.js`
3. `npm test` (if you touched the calculator)
4. `git add -A`
5. `git commit -m "<descriptive message>"`
6. `git push origin HEAD:main`
7. Vercel auto-deploys in ~30 seconds

**Commit message style:** subject line + blank line + body explaining WHY,
not WHAT. Past commits are good examples. Include the cache bump in the body.

Co-authorship trailer is fine but not required.

## Common gotchas

1. **CRLF warnings on `git add -A`** — Windows working tree. Harmless. Files
   are stored as LF in the repo via `core.autocrlf`.
2. **Bumping cache without bumping all 4 references** — `bump-cache-bust.js`
   has OLD_V, NEW_V, RENAMES array, and replacements array. All must agree.
3. **Editing `hero.jpg` without `hero-original.jpg`** — see hero section.
4. **Adding to `scripts/` without running `node scripts/<name>.js` to verify** —
   most scripts are one-shot bulk operations. Test before committing.
5. **The 8 legacy cat breeds** use `<slug>-cat-cost/` directory pattern
   (bengal, british-shorthair, maine-coon, persian, ragdoll, scottish-fold,
   siamese, sphynx). Everything else uses `<slug>-cost/`. Helper:
   `CAT_LEGACY_DASH_CAT` map in `/compare/index.html` and `/find-my-breed/index.html`.
6. **`compare/index.html` and `find-my-breed/index.html` are SELF-CONTAINED.**
   They have inline `<style>` and inline `<script>`. Don't externalize without
   reason. There's a deliberate convention here.

## Honest "stop and let it run" reminders

The audit consensus (GPT + my own analysis) is that the site is in good shape
to start accumulating traffic. The biggest remaining risks are:

- Adding more pages before validating which ones rank
- Hiring a vet reviewer prematurely
- Aggressive affiliate placement before earning trust
- Optimizing pages with no measurement baseline

If a future session asks for "more breeds" / "more pages" / "more guides",
push back unless GA4 + GSC data clearly says where the demand is.

## When in doubt

- **Calculator math change?** Add a test in `tests/calculator.test.js`.
- **New breed page?** Use a generator script in `scripts/`, don't hand-author.
- **Hero photo swap?** Write `hero.jpg` AND `hero-original.jpg`.
- **CSS/JS edit?** Bump cache before pushing.
- **Anything bulk?** Write a one-shot script in `scripts/`, run it, commit
  both the script and the result.
